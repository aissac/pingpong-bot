## Session 2026-04-04: CRITICAL BUG FIX - UTC vs ET Timestamps (14:13 UTC)

### The Bug

**Discovery code was subtracting ET offset from UTC timestamps!**

```rust
// WRONG: Polymarket slugs use UTC, NOT ET
let et_offset = 4 * 3600; // UTC to ET
format!("btc-updown-5m-{}", (now - et_offset) / 300 * 300)
```

**This caused bot to discover markets from 4 HOURS AGO!**

- Current UTC: 14:12 (1775311972)
- Code calculated: `1775311972 - 14400 = 1775297572`
- Result: Markets from 6:10 AM ET instead of 10:10 AM ET

### The Fix

```rust
// CORRECT: Use UTC timestamps directly
format!("btc-updown-5m-{}", now / 300 * 300)
```

**Polymarket slug timestamps are in UTC, not ET!**

### Evidence

| Old Timestamp | New Timestamp | Actual Slot |
|---------------|---------------|-------------|
| 1775297400 | 1775311800 | 10:10-10:15 ET |
| 1775297700 | 1775312100 | 10:15-10:20 ET |
| 1775298000 | 1775312400 | 10:20-10:25 ET |

### Why This Wasn't Caught Earlier

- Bot was running and showing prices
- But those prices were from OLD markets (already resolved)
- No trades triggered because markets were already past their windows
- 15M strategy never triggered because markets were past first 2 minutes

### Lesson Learned

**ALWAYS VERIFY:**
1. Market timestamps match current time
2. Discovery shows CURRENT and NEXT slots
3. Prices are from ACTIVE markets, not resolved ones
4. WebSocket message format (strings vs integers)

**Pattern:**
```
Current UTC: $(date +%s)
Current 5M slot: $(echo "$(date +%s) / 300 * 300" | bc)
```
